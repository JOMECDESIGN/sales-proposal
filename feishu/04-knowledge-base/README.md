# 第④层 · 知识底座(招 18 的工程落地 · PoC)

> **一句话**:把 `案例标杆/`、`模板库/` 等方案资产,沉淀成一个**可检索、带出处**的知识库,让 7 个 Agent(尤其 🛠️ SE / 🏹 Strategist)在做方案时"有据可引"。这是 [招 18 知识底座](../../curriculum/02-方法论-招式卡片.md) 的工程实现。
>
> **状态**:🧪 PoC 设计(尚未 GA)。前三层是飞书原生能力,第④层是**仓外常驻**的知识底座扩展,选型 [`infiniflow/ragflow`](https://github.com/infiniflow/ragflow)。
>
> 配套:[招 18 卡片](../../curriculum/02-方法论-招式卡片.md) · [招 17 逐条应答](../../curriculum/02-方法论-招式卡片.md) · [05-飞书打通-使用手册](../../curriculum/05-飞书打通-使用手册.md)

## 0. 为什么是 ragflow,为什么单独一层

| 选型理由 | 说明 |
|---|---|
| **deepdoc 深度文档理解** | 保结构抽取中文方案里的**表格 / 架构图 / 版式**,不拍扁(标书、PPT 友好) |
| **带出处引用的 RAG** | 检索结果回传"来自哪份方案的哪一段",直接喂招 6/招 11/招 17 |
| **原生飞书 / MCP** | 与本仓飞书生态同源(虽然网页版下我们走直连脚本,见 §2) |
| **仓外常驻** | 它是 ES/MySQL/Redis/MinIO 的重型服务,**不进 Git、不进临时会话**,由你自有主机托管 |

## 1. 网页版约束(决定架构的两条硬边界)

> 你在用 **Claude Code 网页版**。这改变了落地方式,必须先讲清楚:

| 约束 | 来源 | 影响 |
|---|---|---|
| **会话容器临时、闲置回收** | 远程执行环境 | ragflow 的重型栈**不能跑在会话里**;必须部署在你自有的持久主机 |
| **MCP 子进程套出口代理白名单** | 见根 [`CLAUDE.md`](../../CLAUDE.md) 网页版限制条 | 连 ragflow **不走 MCP**,改走**第③层式直连脚本**(HTTP API);这条路 `CLAUDE.md` 已记"不受影响,已验证可用" |

**结论**:网页版下,Claude Code 的角色是**编辑方法论与本设计**;真正的建库与检索,跑在**你自有主机**上,通过 ragflow 的 **HTTP API / Python SDK** 调用。

## 3. PoC 范围与验收(套用本仓 POC 模板)

> 刻意做**小而尖**——证明一件关键事:"过去的标杆能被检索出来、带出处、可被 Agent 引用"。

### 问题陈述
> 本 PoC 验证:**ragflow 能把本仓 3 份标杆建成知识库,针对一个真实售前问题返回正确段落,且带可追溯出处,供 🛠️ SE 在 30 秒内引用。**

### 成功标准(开工前先约定)
| 标准 | 目标 | 测量方式 |
|---|---|---|
| 建库成功 | 3 份标杆全部解析入库,表格不丢 | 在 ragflow 界面抽查"船舶方案"的某张表是否保结构 |
| 检索命中 | 测试问题命中正确段落 | 见下方"测试问题"——返回 Top1 是否为目标段落 |
| 出处可追溯 | 每条结果带"来源文件 + 段落" | 检索 JSON 里有 `document_name` + 定位信息 |
| Agent 可用 | SE 能据此写出带出处的一句应答 | 人工:把结果喂 `sales-engineer.md`,产出是否带出处 |

### 测试问题(固定 3 条,可复跑)
1. 「船舶智能驾驶舱**最强的信任锚点**是什么?」→ 期望命中 **船舶方案 P02:全国首个 CCS 认证**(对应招 11)。
2. 「太空舱方案怎么把**技术能力翻译成价值**?」→ 期望命中 **"十二个『可』"**(对应招 2/招 4)。
3. 「造船智能制造的 **AI 红线**有哪些?」→ 期望命中 **造船知识卡片的红线段落**。

### 范围 In / Out
- **In**:3 份标杆建库;3 条测试问题;1 个直连检索脚本;人工把结果喂 SE 验证。
- **Out(明确不做)**:全量教材入库、飞书机器人问答、多租户、权限、UI 美化、招 17 全流程自动化——**全部留到 PoC 通过之后**。

### 时间线(紧凑,约 1 周)
- D1:自有主机部署 ragflow(Docker Compose),跑通空库。
- D2:灌入 3 份标杆,目视检查解析质量(表格/结构)。
- D3:写直连脚本,跑通 3 条测试问题,记录命中与出处。
- D4:把结果喂 SE,产出带出处应答;对照成功标准打分。
- D5:写结论(GO / NO-GO)+ 下一步范围。

### 决策门(GO / NO-GO)
4 条成功标准**全过 = GO**(进入扩库 + 接招 17 逐条应答);任一不过 = 记录原因,要么调参再试,要么换更轻的方案(见 §6)。

## 4. 落地步骤(在你自有主机执行)

> 以下命令在**你的主机**跑,不在网页版会话里。仅作起步参考,以 ragflow 官方文档为准。

**① 部署**(需 ≥4 核 / ≥16GB / ≥50GB,Docker ≥24)
```bash
git clone https://github.com/infiniflow/ragflow.git
cd ragflow/docker
docker compose -f docker-compose.yml up -d
# 起来后访问 http://<主机IP>:80,创建账号,拿到 API Key
```

**② 建库灌料**
- 新建 Knowledge Base「售前标杆库」,切块模板选适配中文长文/表格的类型。
- 上传(或从 git 同步后上传)`案例标杆/` 的 3 份标杆。
- 等解析完成,**目视抽查**表格与结构是否保留。

**③ 直连检索脚本骨架**(示例,需按你的实例与 SDK 版本调整)
```python
# ragflow_query.py —— 第③层式直连,不走 MCP
# 放在你自有主机运行;凭证只进环境变量/.env,不入库
import os, requests

RAGFLOW = os.environ["RAGFLOW_BASE_URL"]      # 例: http://127.0.0.1:80
API_KEY = os.environ["RAGFLOW_API_KEY"]
DATASET = os.environ["RAGFLOW_DATASET_ID"]    # “售前标杆库” 的 id

def query(question: str, top_k: int = 5):
    r = requests.post(
        f"{RAGFLOW}/api/v1/retrieval",
        headers={"Authorization": f"Bearer {API_KEY}"},
        json={"question": question, "dataset_ids": [DATASET], "top_k": top_k},
        timeout=30,
    )
    r.raise_for_status()
    for c in r.json().get("data", {}).get("chunks", []):
        print(f"[出处: {c.get('document_keyword') or c.get('document_name')}] "
              f"{c.get('content','')[:160]}")

if __name__ == "__main__":
    query("船舶智能驾驶舱最强的信任锚点是什么?")
```
> 接口路径/字段以你部署版本的 API 文档为准;这里只示范"**检索 → 取出带出处的段落**"这一步。

**④ 喂给 Agent**
把脚本输出的"带出处段落",贴进对 `sales-engineer.md` 的提示里,让它按招 18 引用、按招 17 逐条作答。

## 6. 风险与回滚

| 风险 | 应对 |
|---|---|
| 运维偏重(16GB/多组件) | PoC 只在一台主机起;若过重,回滚到更轻的本地 RAG(如纯向量库 + 脚本) |
| 网页版会话碰不到内网 ragflow | 把检索脚本**也放主机跑**,网页版只负责写方法论(架构已假设这点) |
| 中文长文/表格解析不理想 | 调切块模板;或对标杆先做轻度结构化(标题分级)再入库 |
| 凭证泄露 | `RAGFLOW_API_KEY` 只进主机环境变量/`.env`,**永不入库**(同本仓安全约定) |
