## 执行摘要(一页)

船舶智能化的真正瓶颈不在"单船自动化",而在**底层数据割裂、协议各异、系统成孤岛**——结果是体验差、安全管控弱、船岸割裂,且**无法快速复制、无法持续沉淀资产**。

**智能船机交互系统**给出的答案是:在**一套不变的 AI-IoT 数据底座(Yog IOT)**之上,交付**软硬件一体的船端人机协同驾控中枢**,让全船设备数据变成船员能**看、能问、能操、能控**的统一体验;并向岸端延伸为远控、监管与培训能力。

我们靠五个**赢主题**赢单:

1. **软硬一体,一家整合交付,责任单一**——交互硬件 + 软件 + 边缘 AI + 底座,不让客户自己拼。
2. **站在"不变的底座"上**——新设备、新场景在三套契约内低成本扩展,告别"一船一造"。
3. **负责任的 AI 边界**——"看/问/研判"可 AI,"控真实船舶"必须人工确认 + 第三方执行 + 可审计;**负责任 = 可信**。
4. **信创可控**——国产化、本地化部署、数据不外传,面向央企与强监管天然友好。
5. **人机交互基因跨域**——公司智能座舱/人机交互能力下船,交互设计功底是天然壁垒。

> **记忆锚点**:**一个底座,四能力(看·问·操·控),软硬一体。**

## 01 形势与机会 —— 为什么是现在必须决策?(决策者)

- **政策东风**:《智能航运2030行动计划》——2027 年 AI 与航运要素深度融合、关键核心技术突破,2030 年全面掌握并具备谱系化装备供给;"人工智能+交通运输"——2027 年典型场景广泛应用、普及一批智能体。
- **合规红线**:客船智能监控系统**强制安装**(交通运输部海事局《船舶智能监控系统技术指南》);**2026 是智能航运系统化推进的关键期**。
- **行业变革**:竞争逻辑正从"能不能造出来"转向"**智能化水平高不高**"。
- **国产化要求**:央企与监管场景要求**安全可控、本地化、数据不外传**,信创已是硬约束而非加分项。

## 03 我们的答案 —— 你们的东西是什么、怎么入?(业务+技术评审)

### 3.1 定位与两层全景

> **底座负责"设备→数据",产品负责"数据→人机驾控";两层各自演进、互不绑死。**

```
核心产品:智能船机交互系统(软硬件一体)  = 智慧驾驶舱/遥控驾驶的升级·超集
  人 ↔ 船:看(可视化/视频)· 问(AI 跨设备诊断)· 操(IOT PAD 现场闭环)· 控(可控+互锁)
技术底座:Yog IOT(底座不变,三套契约:MQTT Topic / 核心数据表 / REST·WS·SSE)
  设备 ↔ 数据:接入/采集/监控/告警/健康评分/AI 诊断
```

### 3.2 软硬件一体构成("一体"指什么)

| 维度 | 构成 |
|---|---|
| 边缘硬件(心脏) | **AI+IOT 智能体网关 V2**:一台终端汇聚全船设备;接口 串口/网口/CAN/485/IO |
| 交互硬件 | 驾驶台大屏 · 机舱屏 · 操控台 · **IOT PAD 移动工作台** · 〔待填:语音阵列/冗余主机型号〕 |
| 交互软件 | 可视化 · 实时监控 · 告警确认 · AI 问答 · 视频监控 · 操控/驾控界面 |
| 边缘 AI | 本地诊断/异常/根因 + 智能问答(可对接 MindDuck/客户工作流) |

> 差异点话术:**一家整合交付,责任单一**——不是让客户买一堆零件自己拼。

### 3.3 四能力主线(带客户价值)

| 能力 | 它做什么 | 客户价值 |
|---|---|---|
| 统一可视化驾控 | 全船状态/在线离线/异常劣化汇成驾驶台、机舱、大屏视图 | 一屏掌握全船,无需多系统切换 |
| 即插即用接入 | 按类型+协议配模板,自助解析+端口映射;识别=历史记忆+指纹匹配 | 设备上线自动配置,摆脱资深工程师强依赖 |
| 告警分级 | P0 紧急 / P1 异常预警 / P2 运行提示 | 减少无效打扰,突出关键风险 |
| 健康分机制 | 熔断红线 + 短板扣分 + 趋势 → 可解释健康分 | 从"坏了再修"到预测性维护 |
| AI 跨设备诊断 | 结合航行状态+标签+健康趋势+关联设备研判 | 不懂协议也能定位;根因+主动预判 |
| 视频监控 | 视频源/矩阵/单路/告警取证 | 事件可视化、可追溯 |
| 可控设备+互锁 | 可控设备权限与互锁体系;统一控制入口 | 控制可分级、可审计(见 05/附录B) |
| IOT PAD 现场闭环 | Pad 整合设备+视频+控制,现场查看/确认/操作 | 移动工作台,现场完成处置 |

### 3.4 客户路径分级(招13,降低门槛)

| 起点 | 范围 | 适合 |
|---|---|---|
| 小 | 单舱单屏可视化监控起步 | 先验证、预算有限 |
| 中 | 监控 + 告警 + 健康评分 + 驾控 | 船端驾控中枢 |
| 大 | 全船软硬一体 + 遥控 / 船岸协同 | 旗舰/示范船、岸基中心 |

## 05 为什么是我们 —— 为什么不选竞争对手?(全体)

### 5.1 四维差异化对比矩阵(招8,对比但不贬低)

| 维度 | 智能船机交互系统 | 传统集成方案 |
|---|---|---|
| 交付模式 | **软硬空间一体化,一家整合交付,责任单一** | 软硬碎片、多家拼装、责任分散 |
| 扩展逻辑 | **站在不变底座的三套契约上,低成本加设备/场景** | 一船一造、重复改造、改一处动全身 |
| AI 形态 | **AI 原生**(语音+大模型+跨设备诊断)且**负责任边界** | 传统人机界面,AI 事后挂载或越界 |
| 可控可信 | **信创可控、本地化、数据不外传** | 依赖国外生态、数据外传顾虑 |

### 5.2 负责任的 AI 边界 = 可信(差异化升维)

我们对"控"的态度是差异点,不是短板:**"看/问/研判"可以 AI,"控真实船舶"必须人工确认 + 第三方执行 + 可审计**。完整边界见 [附录 B]。这对监管与安全责任主体是**减分项变加分项**。

### 5.3 原生合规 + 生态背书

- **按监管规则设计,不是事后适配**(CCS 认证底子)。
- 〔待填:高校/海事局/校企生态名单——重点是名单本身〕。

## 06 合作路径 —— 第一步怎么迈?(尤其决策者)

### 6.1 三步走 + 退出机制(招14)

| 步 | 内容 | 验收口径 |
|---|---|---|
| 第一步 | 单船/单舱概念验证(可视化监控起步) | 稳定运行 30 天、数据联通率 >90% |
| 第二步 | 监控告警 + 健康评分 + 驾控 / 岸端接入 | 客户可用的监管与运维入口、满足报建接口 |

> **退出机制**:任一步验收不满意,可随时协商,不影响后续模块——**小步快跑、低风险**。

## 附录(主材料精简,深度内容按受众分层调用)

### 附录 A · 技术架构(给技术评审)
- **五层消息驱动管道**:现场设备 → 采集端(protocol-router/telegraf/device-ai-identifier)→ **EMQX MQTT(脊椎)** → yogee 服务端 → PostgreSQL/TimescaleDB/Redis → 前端/大屏/IOT PAD/AI → Loki/Grafana。
- **三套不变契约**:① MQTT Topic 命名空间(`/data/device` `/iot/status` `/iot/error` `/iot/router/bind` `/iot/identify/*`);② 核心数据表(`devices`/`device_alarms`/`device_registry`/`device_health_score_*`/`port_cards*` 等);③ 接口形态(REST/WS/SSE/Swagger)。
- **两条业务流**:已知设备自动绑定(无 AI);未知设备 AI 辅助识别 + 人工确认 + 热重载即插即用。
- **客户必须前置提供**:Modbus 厂家寄存器表(量纲/倍率/报警位/通讯参数)。

### 附录 B · 控制与 AI 责任边界(给监管/安全评审)
- 不替代第三方船控(真实底层控制 + 安全联锁由第三方负责);AI 仅辅助、不替代人工与法定责任主体;AI 不直接执行真实底层控制;真实/仿真/演示数据显式区分;全过程留痕可审计。

### 附录 C · 国产化 / 信创(给信息部门/央企)
- 分层算力(边缘 + 本地服务器 + AI 推理);国产芯片(海光/鲲鹏/飞腾/龙芯/昇腾/寒武纪/沐曦/RK3588);信创数据栈(PostgreSQL/TimescaleDB/向量库,与底座同栈);本地私有化大模型 + RAG。〔待填:我方实际适配清单〕
