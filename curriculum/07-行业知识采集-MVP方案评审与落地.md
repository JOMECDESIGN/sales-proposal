# 行业公开知识采集 · MVP 方案评审与落地

> **为什么有这一篇**:售前的弹药是「行业情报」——客户底细、政策风向、竞品动作、技术趋势。
> 这一篇评审一套「行业公开知识采集 MVP」的可行性,给出**收敛后的落地路径、合规红线与元数据规范**,
> 让采集来的素材能**直接、可信、可溯源地**喂进本仓库的 7 个 Agent,服务方案、招投标、客户拜访与趋势研判。

---

## 〇、这套东西要解决什么

售前四个高频场景,都卡在「找料慢、料不新、料不可信」:

| 场景 | 需要的料 | 没有采集库时的痛 |
|---|---|---|
| 写售前方案 | 行业趋势、技术白皮书、案例 | 临时搜、东拼西凑、深度不够 |
| 招投标技术文件 | 政策依据、标准、采购公告 | **引用不可溯源 / 内容过期 → 废标风险** |
| 客户拜访前情报 | 客户官网、组织、近期动态 | 拜访前临时抱佛脚 |
| 行业趋势研判 | 竞品动作、展会、政策 | 信息零散、无沉淀 |

**目标产物**:一个「带引用回答」的售前问答助手 + 一个可持续更新的行业素材库。

---

## 一、被评审的方案(原始设想)

```
采集(Crawl4AI / Firecrawl) → 清洗(Trafilatura) → Markdown/JSON 入库
   → 知识库(RAGFlow / Dify) → 售前问答助手
```

优先抓 7 类内容:**客户官网、央企采购公告、船舶智能化政策、智能座舱案例、AI-IoT 技术白皮书、竞品官网、展会资料**。

---

## 二、总评:方向对,但要「先窄后宽、先合规后规模、先溯源后问答」

| 维度 | 评分 | 说明 |
|---|---|---|
| 价值 / 选题 | ⭐⭐⭐⭐⭐ | 直击售前四大场景,值得做 |
| 技术可行性 | ⭐⭐⭐⭐ | 主流 RAG 数据流,成立 |
| MVP 收敛度 | ⭐⭐ | 偏「全都要」,是工程蓝图不是 MVP,需砍 |
| 合规设计 | ⭐⭐ | 央企采购、竞品官网两类是雷,当前无护栏 |
| 溯源 / 时效设计 | ⭐ | **缺失,招投标场景的硬伤** |

**一句话**:做,但当前是「全都要」的蓝图。最大风险不在技术,而在 **① 合规边界** 与 **② 溯源时效**。

---

## 三、架构评审:管线成立,但有工具冗余

这条主线是业界标准 RAG 数据流,没问题。但**三处叠了重复能力**:

| 环节 | 原方案 | 问题 | 建议 |
|---|---|---|---|
| 采集 + 清洗 | Crawl4AI **+** Firecrawl **+** Trafilatura | 三者正文抽取能力重叠 | **抓取选一个**:本地批量 + 商用合规(Apache-2.0)→ **Crawl4AI**;省事、自然语言驱动 → Firecrawl(自托管 AGPL-3.0,商用先看授权)。**Trafilatura 只作二次质检 / 正文兜底 / 元数据(发布日期、作者)抽取** |
| 知识库 | RAGFlow **+** Dify | 定位不同,非二选一也非简单叠加 | **RAGFlow 做检索内核**(DeepDoc 解析 PDF / 表格强,白皮书与招标文件必需);**Dify 做上层助手 App + 工作流 + 多模型编排**。两者互补,但 **MVP 先上 RAGFlow** 一个 |

> 冗余不是错,是 MVP 阶段的成本。每多一个组件 = 多一份部署、反爬维护、版本升级负担。

## 四、7 类内容分级(全篇最该先想清楚的地方)

按「售前价值 × 抓取难度 × ⚠️合规风险」分级,不同内容差别极大:

| 内容类型 | 价值 | 难度 | ⚠️合规风险 | 处置 |
|---|---|---|---|---|
| AI-IoT 技术白皮书 | 高 | 中(多 PDF) | 低 | ✅ **首批**,需强 PDF 解析 |
| 船舶智能化政策 | 高 | 中(政府站反爬) | 低 | ✅ **首批**,优先官方 RSS / 公开接口 |
| 智能座舱案例 | 高 | 低 | 低 | ✅ **首批** |
| 客户官网 | 中(拜访情报) | 低、量小 | 低 | ✅ 礼貌低频抓 |
| 展会资料 | 中 | 中(gated / PDF) | 低 | 🟡 二批 |
| 央企采购公告 | 高(招投标用) | 高(登录 / 反爬 / 平台分散) | 🟠 中:公开可读,但平台 ToS 多禁批量抓 | 🟡 **谨慎**,优先官方公示,控频,留来源 |
| 竞品官网 | 中 | 中 | 🔴 高:robots / ToS + **反不正当竞争**顾虑 | 🔴 **不自动化批量**,改人工辅助单页、小量、留痕 |

## 五、合规红线(做之前先立规矩)

> 这套是要服务**招投标技术文件**的——AI 答错或引用了过期 / 不可溯源内容进了投标书,代价远高于省下的调研时间。**合规与溯源不是加分项,是准入门槛。**

- 🔴 **只采公开内容**:不碰需登录、付费墙、明确禁爬(robots.txt / ToS)的内容。
- 🔴 **竞品官网不批量自动化**:改人工辅助、小量、留痕;避免反不正当竞争争议。
- 🟠 **央企采购公告控频留痕**:优先官方公示渠道,限速,记录来源 URL 与抓取时间。
- 🟠 **礼貌抓取**:遵守 robots、限速、标识 UA;客户官网尤其低频。
- 🔴 **入库即标信源**:任何内容必须带来源,不可「裸文本」入库(见第六节)。

## 六、方案缺的 4 个关键件(不补,招投标场景会翻车)

### 1. 溯源元数据(最重要)
每条入库内容必须带以下字段,问答助手**强制引用、给出处**:

| 字段 | 说明 | 例 |
|---|---|---|
| `source_url` | 来源 URL | https://... |
| `crawled_at` | 抓取日期 | 2026-06-25 |
| `published_at` | 原文发布日期(Trafilatura 抽) | 2026-03-01 |
| `authority` | 信源权威级别 | 官方 / 行业媒体 / 企业自述 |
| `domain` | 业务域 | 船舶运营 / 造船制造 / 智能座舱 |
| `content_type` | 内容类型 | 政策 / 白皮书 / 案例 / 公告 |

> 没有这层,生成内容**不能进投标书**。

### 2. 时效 / 增量更新
政策、采购公告**时间敏感**。需定时增量爬 + 变更检测,检索时对过期内容降权或标注。
**一次性全量抓 = 三个月后知识库全是旧闻。**

### 3. 幻觉护栏
售前 / 投标场景**宁可答「无依据」也不能编**:检索不到就拒答 + 强制引用 + 关键结论人工复核。

### 4. 去重与噪音过滤
同一政策被多站转载会污染检索。需 URL / 正文指纹去重。

## 七、收敛后的 MVP:一条窄缝跑通

不要 7 类内容、3 个采集器、2 个知识库一起上。真正的 MVP:

> **选 1 个领域(建议「船舶智能化政策 + 案例 + 白皮书」,对应元启晟船舶标杆)→ Crawl4AI 抓 + Trafilatura 质检 → 出带元数据的 Markdown → RAGFlow 建库 → 一个能「带引用回答」的问答接口。**

跑通这条闭环(约 50–100 篇素材),验证「检索质量 + 引用可信 + 真能帮到 SE / 方案」后,再横向扩内容类型、再叠 Dify 做工作流、再碰采购公告 / 竞品这类高风险源。

### 分阶段落地清单

| 阶段 | 目标 | 内容范围 | 工具 | 验收标准 |
|---|---|---|---|---|
| **P0 闭环验证** | 跑通采集→入库→带引用问答 | 船舶政策 + 案例 + 白皮书(50–100 篇) | Crawl4AI + Trafilatura + RAGFlow | 抽 10 个真实售前问题,答案**都带可点击出处**且事实正确 |
| **P1 提质** | 解决时效与噪音 | 同上 + 增量更新 | + 定时增量 + 去重 + 元数据校验 | 政策更新 24h 内入库;无重复污染 |
| **P2 扩域** | 横向铺内容 | + 智能座舱、客户官网、展会、造船制造 | 同栈 | 各域各 ≥30 篇,问答覆盖四场景 |
| **P3 产品化** | 接入工作流与团队 | 全量 | + Dify 工作流 + 飞书分发 | 助手嵌入 Agent 链路与飞书(见第八节) |
| **P4 高风险源** | 谨慎接入 | 央企采购公告(竞品仅人工) | 控频 + 留痕 | 合规复核通过后再上 |

## 八、与本仓库怎么接(关键:长在现有产线上,而非孤立系统)

- **入库位置** → 新建 `行业素材库/`(或并入 `案例标杆/`),Markdown 落盘,天然衔接 `06-排版篇` 的「Markdown → PDF」出片链路。
- **消费方(喂给哪个 Agent)**:

| 售前场景 | 主导 Agent | 用采集库做什么 |
|---|---|---|
| 技术情报 / 白皮书 / 案例 | 🛠️ Sales Engineer | 能力→价值翻译时取证;Demo / POC 找参照 |
| 客户情报 / 干系人 | 🗺️ Account Strategist | 拜访前情报、组织与采购路径 |
| 行业趋势 / 政策 / 竞品 | 🧭 Product Manager | 趋势研判、平台演进对标 |
| 招投标技术文件 | 🏹 Proposal Strategist + 🐑 Project Shepherd | 政策依据、标准引用(强制溯源) |

- **飞书层(复用 `05-飞书打通`)**:
  - 第②层(`feishu/02-doc-sync`)做素材分发:Markdown ↔ 云文档同步。
  - 第③层多维表格做「采集源看板」:信源 / 业务域 / 更新时间 / 状态 / 合规级别。

|---|---|---|---|

## 十、结论

**做,但先窄后宽、先合规后规模、先溯源后问答。**

1. 砍到一条窄缝(船舶政策 + 案例 + 白皮书)先跑通闭环;
2. 采集器只用 Crawl4AI,Trafilatura 做质检,RAGFlow 做库;
3. **从第一天就带溯源元数据 + 强制引用**;
4. 央企采购、竞品官网放到最后、谨慎处理;
5. 产物长在 7 个 Agent 与飞书产线上,不做孤岛。
