## 一、一句话定位

> 以 **AI-IoT 数据与智能底座**为核心,面向智能船舶**远控场景**,集**船舶多源数据接入、岸基辅助研判、智能预警、任务反馈、岸基监管、模拟训练、培训复盘、运维审计、展示汇报**于一体的**综合性岸基智能支撑平台**。

它**不是**单一软件、不是设备采购、不是展厅装修;是**数据平台 + 智能分析 + 空间装备 + 交互体验 + 场景展示**的一体化示范工程。**建设重点明确:不替代船端既有控制系统或第三方船控系统**,而是建岸基的"感知—研判—监管—训练—审计"能力。

   ├─ 船端:驾驶舱/机舱/IOT 网关V2 …………… 已有文档

## 三、总体架构:17 个子系统 + 五条链路

方案把软件拆成 **17 个子系统**,可归为六组:

| 组 | 子系统 | 作用 |
|---|---|---|
| 接入 | IOT 集成平台、船舶设备数据集成、远控室设备数据集成、模拟舱数据集成 | 多协议接入→解析→标准化→分发(NMEA/Modbus/CAN/PLC/MQTT/HTTP/WS) |
| 智能 | AI 辅助分析与智能预警、**智能机舱感知增强智能体**、**声音量化故障诊断** | 质量识别/异常检测/趋势预警/告警研判/根因/处置建议 + 感知增强 + 声纹诊断 |
| 控制 | 第三方船控接口适配与状态反馈、控制任务与反馈管理 | 任务登记→权限/参数校验→第三方接口调用→反馈接收→日志(**不执行真实底层控制**) |
| 监管展示 | 数据大屏、塔台任务大屏、岸基监管 | 总体态势 / 任务流转 / 综合监管告警处置 |
| 训练 | 模拟舱模拟、塔台模拟舱控制、模拟舱硬件交互、培训 | 场景仿真 / 异常注入 / 硬件交互采集 / 课程考核复盘 |
| 运维 | 系统运维支撑 | 权限/日志审计/运行监控/备份/知识库 |

**五条链路**:数据链路、控制链路、智能预警链路、展示监管链路、模拟训练链路、运维支撑链路——各链路"接口化、状态化、日志化、可追溯"。

## 四、新增能力维度(相对我们已有文档,这份补了什么)

1. **智能机舱感知增强智能体**:全生命周期数据贯通治理 + 感知可靠性增强 + **盲区状态重构与虚拟感知**(传感器健康度/数据置信度/虚拟感知值/虚实一致性)。
2. **声音量化故障诊断**:持续积累不同设备/工况/异常的声音数据 → **声学故障知识库**(异常声纹/异常频段/趋势偏离/风险等级)。
3. **岸基辅助研判**:融合定位/航向/雷达/AIS/电子海图/机电状态 → 环境态势、目标风险(CPA/TCPA/DCPA)、设备可用性综合判断。
4. **智能预警 + 告警研判**:规则引擎 + AI;告警聚合/去重/分级/影响范围/根因;输出**置信度 + 处置建议 + 复盘知识**。
5. **岸基监管 + 三大屏**:数据大屏(总览)、塔台任务大屏(任务流转)、岸基监管(综合监管处置)。
6. **模拟训练/培训复盘**:模拟舱仿真 + 异常注入 + 硬件交互采集 + 课程/考核/复盘;对标国际模拟教学(Conning/Overhead Display、AI 虚拟值班员、虚拟望远镜、扩展故障系统、新一代 ECDIS/S-100)。
7. **空间与装备一体化示范载体**:胶囊式模拟驾驶舱、模拟/远控/培训驾控台、U 型沉浸式巨幕屏、远控信息大屏、远程会议大屏、控制设备机柜——"软硬件一体"在岸基的最强体现。

## 五、⭐ 安全与责任边界(本案例最值钱的可复用原则)

这是把 AI 放进**安全关键的船舶控制场景**时的**负责任边界**,直接呼应我们之前标记的"控制/边缘AI 口径"风险——这份方案给了教科书级答案,**所有涉控方案都应照搬这套话术**:

- **平台不替代第三方船控系统**:真实船舶底层控制、推进、舵机、主机、电气动力执行及**安全联锁**,由**第三方船控系统**负责;平台只做**控制任务管理、接口适配、权限校验、执行反馈接收、日志审计、复盘**。
- **AI 仅辅助、不替代**:AI 输出是**辅助研判和处置建议**,不替代人工确认、船端控制系统、设备厂家诊断或**法定安全责任主体**。
- **AI 不得直接生成/执行真实船舶底层控制指令**:涉真实控制动作的建议,必须进入人工确认 → 任务管理 → 第三方接口调用 → 状态反馈流程。
- **真实/仿真/演示数据必须显式区分**:界面、日志、展示脚本都要标清,**避免对真实远控能力和控制责任产生误解**。
- **人工确认闭环 + 全过程留痕**:任务发起→校验→下发→反馈→异常→复盘全链路日志可追溯。

> 一句话:**"看/问/研判"可以 AI,"控/处置真实船舶"必须人工确认 + 第三方执行 + 可审计**。这既是合规底线,也是差异化(负责任 = 可信)。

## 六、国产化 / 信创路线(面向央企·强监管客户的差异化弹药)

方案要求**国产化、安全可控、本地化部署、内网运行、离线推理、数据不外传**。分层算力 + 国产选型清单(招投标可直接引用,**需确认我方实际适配清单**):

- **分层算力**:边缘算力(多协议接入/缓存/边缘解析/轻量规则)+ 本地服务器算力(数据服务/存储/监管/多屏)+ AI 推理算力(异常识别/声学分析/虚拟感知/RAG 问答)。
- **芯片/算力**:CPU 海光/鲲鹏/飞腾/龙芯;AI 加速卡 昇腾 Ascend/寒武纪 MLU/沐曦;边缘 飞腾/瑞芯微 RK3588 国产工控机。
- **数据层**:关系库 PostgreSQL · 时序库 TimescaleDB · **向量库(RAG)** · 文件存储——**与 Yog IOT 底座同栈**。
- **AI 层**:本地私有化大模型 + Embedding + 语音识别/合成 + RAG 检索增强;**输出可追溯、可人工确认**。
- **原则**:核心环节可控、关键能力可替代、系统接口开放、数据资产可沉淀、运维可持续。

## 七、接口体系(14 类接口,落在底座契约思路上)

接口设计原则:**来源明确、对象明确、参数明确、结果明确、异常明确**;统一命名/字段/返回格式/错误码/调用日志。14 类接口含:IOT 边缘数据、实时数据、历史数据、设备状态、控制指令、控制反馈、远控输入、任务管理、告警事件、AI 分析、知识库、模拟舱数据、培训数据、日志审计。

- **实时类**:周期上报 / 订阅推送(对应底座 MQTT + WS)。
- **控制类**:请求编号 + 任务编号 + 反馈状态**闭环管理**(ACK/校验/执行中/成功/失败/超时)。
- **AI 类**:分析编号 + 输入范围 + 结果 + **置信度 + 人工确认状态**。

> 衔接判断:这套接口体系**完全可由 Yog IOT 三套契约扩展承载**(MQTT Topic / 数据表 / REST·WS·SSE),岸基平台是底座契约的"业务面放大",不破坏"底座不变"。
