## 1. 系统架构要求(RFP 3.2)

| # | RFP 要求 | 参考实现(已落地) | 满足度 / 应答动作 |
|---|---|---|---|
| 1.1 | 自适应**容器化与编排**,组件标准化部署、弹性调度 | 全量服务 **Docker 容器化**,compose 编排,镜像 Tag 基线管理 | ✅ 直接复用部署架构图 + compose 基线 |
| 1.2 | **云/岸/船 三级**弹性架构;云端负责物联网接入转发 | 已实现**船/岸两级**一致架构(岸端不装 Telegraf,船端 MQTT 同步) | 🔴 **缺云端层**。应答:补"云端 = 分布式消息中间件(EMQX 集群)+ 标准 API 网关"设计,工作量计入报价 |
| 1.3 | 船端**双活主备**高可用 | 船端双机 **keepalived VIP + nginx 反代**,中间件主从同步,etcd 服务发现;VIP 切换演练通过 | ✅ 有 VIP 漂移切换实测记录。**坑已知**:切换时长连接短暂中断→已有"nginx 超时调优+前端自动重连"对策 |
| 1.4 | 岸端**超融合**架构,支持与船载端映射部署 | 岸端结构与船端一致,承担日志/视频大文件上岸集中管理 | 🟡 "超融合"为部署形态用语,需在方案中明确以超融合一体机承载岸端容器集群 |
| 1.5 | 船端多协议适配:**Modbus、OPC UA、MQTT、CoAP、HTTP/HTTPS** | 已支持 **Modbus RTU/TCP、NMEA0183、CAN、MQTT、RTSP、HTTP**(Telegraf 采集) | 🟡 **缺 OPC UA、CoAP**。应答:OPC UA 经 Telegraf opcua 插件/独立采集服务补齐;CoAP 加轻量网关。列为"可扩展协议",承诺交付前补全 |
| 1.6 | 混合存储与数据治理(整合/清洗/标准化) | 多源数据按 **ISO19848-2024** 标准化输出,字段映射表受控 | ✅ ISO19848 字段映射模板为现成组织资产 |
| 1.7 | 全面适配 **Linux**,兼容主流发行版 | 运行平台 Linux(交付目标 Ubuntu 22.04 见 5.x) | ✅ |
| 1.8 | 多类型数据库:**时序、关系、数仓、文件类** | 已用 **Redis / MySQL / PostgreSQL** 主从 | 🟡 **未见独立时序库**;RFP 3.3.3 点名 **InfluxDB**(配置项清单亦列 influxDB)。应答:补 InfluxDB 承接时序测点,MySQL/PG 承接关系/对象,文件类落对象存储 |
| 1.9 | 消息清洗、转发(过滤/转换/按规则流转) | Golang 服务按设备/业务维度路由,EMQX Topic 精准分发,消息体含设备元信息 | ✅ |

| 1.11 | **船岸数据同步**(双/单向、增量、一致性校验) | 船端 → 岸端 EMQX 实时同步,大文件上岸,岸端页面一致性验证通过 | ✅ 有船岸联试记录。补"增量同步 + 一致性校验"机制说明 |
| 1.12 | 云端:分布式消息中间件高吞吐路由 + 标准 API + 设备全生命周期管理(接入/认证/监控/升级) | 设备接入/认证(MQTT ACL)有,**生命周期管理(配置升级、监控)未成体系** | 🟡 与 1.2 一并补云端能力,突出设备"接入-认证-监控-配置升级"闭环 |

## 3. 性能要求(RFP 3.4)—— 须实测背书

| # | RFP 指标(船端局域网) | 参考实现现状 | 满足度 / 应答动作 |
|---|---|---|---|
| 3.1 | 设备列表查询响应 **≤200ms** & **≥200 QPS** | 系统联试压测覆盖,具体值未在报告量化 | 🟡 **缺量化实测值**。应答:在投标前补一轮基准压测,出具数据;验收按系统联试压测口径 |
| 3.2 | 实时推送:采集→推送 **≤500ms**;**≥50 订阅**;**QoS1 送达率 ≥99.9%**;**≥200 并发连接**;峰值 **≥100 条/秒** | EMQX 架构天然支撑,送达率/延迟未量化 | 🟡 同上,补 EMQX 压测报告(EMQX 单机十万级连接,指标留足余量) |
| 3.3 | 历史分页(≤1000 条/页)**≤1000ms** & **≥50 QPS** | 数据中心支撑,未量化 | 🟡 补分页查询基准测试 |
| 3.4 | **72h@80QPS** 连续压测,无内存泄漏、无连接池耗尽 | 系统联试有长稳运行验证 | 🟡 补 72h 长稳压测专项报告(达标证据) |
| 3.5 | 异常容错(无效参数/超范围/高频重复)不崩溃 | 异常/边界用例(超时、断连、格式异常)已覆盖 | ✅ 有缺陷闭环记录 |
| 3.6 | 双活主备异常恢复全流程 | VIP 切换演练通过,恢复可用 | ✅ 量化切换窗口即可 |

> **关键动作**:性能是本标最大的"证据缺口"——参考实现走的是"系统联试压测替代独立测试"的裁剪路线,**没有逐指标量化值**。Sales Engineer 须组织一轮针对性基准压测,把 6 项数字坐实,否则技术标性能章节是空的。
