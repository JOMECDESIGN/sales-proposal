## 一、一句话定位

> **智能船机交互系统**:软硬件一体的**船端人机协同驾控中枢**——把全船设备数据(由 Yog IOT 底座标准化喂上来)变成船员能**看、能问、能操、能控**的统一交互与驾控体验。

它不是一块屏,也不是一套软件,而是 **"交互硬件 + 交互软件 + AI + 底座数据"** 的整体。卖点的主语是"**一体**":软硬件由我们整合交付,不是让客户自己拼。

## 二、它在产品全景中的位置(两层关系)

```
┌──────────────────────────────────────────────────────┐
│  核心产品:智能船机交互系统(软硬件一体)  ← 本文          │
│  = 智慧驾驶舱 + 遥控驾驶 + 更多船机交互/操控场景(超集)     │

├──────────────────────────────────────────────────────┤
│  技术底座:Yog IOT(底座不变,三套契约)                    │
│  设备 ↔ 数据:接入/采集/监控/告警/健康评分/AI 诊断          │
└──────────────────────────────────────────────────────┘
```

**分工**:底座负责"把异构设备数据标准化、可订阅地喂上来";交互系统负责"把数据变成人能驾驭的体验与操控"。底座变化在契约内部,交互系统在契约之上做产品——**两层各自演进、互不绑死**,这正是"底座不变"对产品迭代的价值。

**与智慧驾驶舱的关系(超集)**:智慧驾驶舱是这套系统在"船端驾控中枢"这一场景的体现;智能船机交互系统在其之上**外延**——把遥控驾驶、机舱交互、以及更广的船机交互/操控场景纳入同一产品框架。〔待填:超集相对驾驶舱新增的具体场景清单〕

## 四、核心能力(已知部分实写,带客户价值)

| 能力 | 它做什么 | 客户价值 |
|---|---|---|
| 统一可视化驾控 | 把全船设备状态/在线离线/异常劣化汇成驾驶台、机舱、大屏视图,全域态势实时可视 | 一屏掌握全船,无需多系统切换,替代割裂的厂家面板 |
| 模版化即插即用 | 按设备数据/协议类型设模板,接入时自动/手动选模板→自动协议解析+端口映射;识别=历史接入记忆+指纹匹配 | 设备扩充沦为"流水线作业",摆脱资深工程师强依赖 |
| **数据系统化打标** | 给每台设备/传感器打"系统分类"(动力/配电)+"位置分类"(主推舱/驾驶舱)标签 | AI 从物理位置+系统层级判断根因来自哪个舱室、连带哪些上下游 |
| **船舶状态识别(工况上下文)** | 由对地航速/转速/航向变化率/侧推/对水航速,自动识别四阶段:进出港机动·停泊系泊·正常航行·无动力漂航 | 给 AI 最核心的"船舶环境上下文",数据结合工况才有意义,赋予航行环境感知力 |
| 设备离线状态识别 | "离线"=IOT 是否收到数据 + 是否**应该**有数据 的综合判断(按工况/上游主设备启停) | 杜绝 CAN 设备静默被误判离线(如主机停机时不需要水泵发数据) |
| **告警分级** | P0 紧急(触物理红线,立即停机/干预)· P1 异常预警(超时防抖的趋势恶化)· P2 运行提示(后台静默记录) | 90% 时间免无效弹窗骚扰,1% 紧急时刻拿到最高优先级阻断提醒 |
| **健康分机制** | 熔断红线 + 短板扣分 → 每台设备实时 0-100 健康分;时序=AI 的"故障趋势"细则 | 一眼看设备退化;迈向预测性维护的关键 |
| **AI 跨设备融合诊断** | AI 助手挂 3 个 MCP 工具(①航向状态 ②设备分类标签与数据 ③健康分细则与趋势),融合关联系统报错+环境+健康下滑研判 | 例:由主推高温+发电机大负载,精准诊断"螺旋桨卡死"而非单纯冷却故障 |

| **视频监控** | 视频源列表、位置筛选、矩阵/单路查看、关联设备与告警取证 | 告警可视化取证,事件可追溯 |

## 五、它如何站在 Yog IOT 底座上(消费三套契约)

这是技术评审最关心的衔接点,**全部落到底座既有契约,不另起炉灶**:

- **读数据**:订阅 MQTT `/data/device`、`/iot/status`、`/iot/error`,或经服务端 REST/WS 取设备列表与详情 → 驱动可视化与监控页。
- **告警闭环**:经服务端 REST 确认告警(写 `device_alarms`),经 WebSocket 收实时告警/识别事件。
- **健康评分**:经 REST/WS 取 `device_health_score_history` 与实时推送 → 健康看板。
- **AI 交互**:经 SSE / 流式代理对接服务侧 AI 工作流(MindDuck/客户)→ 问答与诊断逐段输出,降低前端等待感。
- **新增设备**:底座"AI 辅助识别 + 人工确认 + 热重载"流程产生的动态台账,交互系统直接消费,**新设备即插即现**。

> 一句话:**交互系统只面向契约编程**——底座换实现、加设备,交互系统不改;交互系统加界面、加场景,底座不动。

## 六、控制与 AI 责任边界(涉控方案必照搬)

这是把 AI 放进**安全关键船舶控制场景**的负责任边界,也是第四节"控制/边缘AI 口径"风险的**正式答案**(源自岸基远控项目实践)。**凡涉及"控"的方案,这套话术原样照搬**:

- **不替代第三方船控**:真实船舶底层控制、推进、舵机、主机、电气动力执行及**安全联锁**,由第三方船控系统负责;本产品做**控制任务管理、接口适配、权限校验、执行反馈接收、日志审计、复盘**。
- **AI 仅辅助、不替代**:AI 输出是**辅助研判与处置建议**,不替代人工确认、船端控制系统、设备厂家诊断或**法定安全责任主体**。
- **AI 不直接执行真实底层控制**:涉真实控制动作的建议,必须走 人工确认 → 任务管理 → 第三方接口调用 → 状态反馈 闭环。
- **真实/仿真/演示数据显式区分**:界面、日志、演示脚本都标清,避免对真实远控与控制责任产生误解。
- **全过程留痕**:发起→校验→下发→反馈→异常→复盘,链路可审计。

> 一句话:**"看/问/研判"可以 AI,"控真实船舶"必须人工确认 + 第三方执行 + 可审计**。负责任 = 可信,这本身就是差异点(互锁/边界是卖点不是短板)。
