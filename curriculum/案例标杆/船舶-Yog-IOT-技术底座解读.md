# 产品技术底座 · 船舶运营域 · Yog IOT 深度解读

> **业务领域**:船舶智能化 / 智能航运(元启晨星 **运营域**)。与 [船舶方案-架构思路.md](船舶方案-架构思路.md) 互补——那份讲**售前叙事弧**,这份讲**底层产品技术**。
> **素材来源**:《Yog IOT 项目技术说明书》客户版 V1.0(2026-06-23)——船舶设备接入 / 边缘采集 / 实时监控 / 告警 / 健康评分 / AI 诊断。
> **定位**:这是产品技术的**底座**。后续所有船舶运营域方案在此之上拓展,**底座本身不变**——"不变的"不是代码,而是本文第四节那三套**契约**。
> **怎么用**:售前讲技术能力、做技术发现、和第三方谈对接时,从这里取弹药(对应 🛠️ Sales Engineer、📐 UX Architect、🐑 Project Shepherd)。

---

## 一、本质:卖的不是"采集",是"接入与监控的标准化底座"

原文第 1 节自己点破:**"项目目标不是单纯完成单个设备的数据读取,而是建立一套可扩展、可运维、可追溯的船舶设备接入与监控平台。"**

这决定了叙事主语。产品价值锚点是三个词——**可扩展、可运维、可追溯**,而不是"我能读 NMEA / CAN / Modbus"。后者只是手段,前者才是客户买单的理由。**讲方案时主语必须是"平台 / 底座",协议清单是论据而非主张。**

---

## 二、底层逻辑:一条"消息驱动"的五层流水线

```
现场设备 / PLC / 串口服务器 / CAN
  → 采集端(protocol-router / telegraf-collector / device-ai-identifier)
  → EMQX MQTT 消息总线          ← 真正的"脊椎"
  → yogee-iot-golang 服务端中台
  → PostgreSQL / TimescaleDB / Redis
  → 前端大屏 / 第三方 / AI 诊断
  → Loki / Grafana 运维监控
```

整套设计最核心的一次抉择是 **"消息驱动"而非"调用驱动"**:采集与业务之间用 MQTT 解耦,带来三个直接后果——

1. **多业务可订阅同一数据源**:大屏、告警、健康评分并行消费 `/data/device`,互不阻塞。
2. **采集端与服务端独立部署、独立升级**:这是"船端自治 + 岸端汇聚"船岸协同架构成立的前提。
3. **扩展代价低**:新增消费方 = 新增订阅者,不动既有链路。

这正是"底座不变"的技术含义:**不变的是消息契约,变化发生在契约内部。**

## 三、五层模块与"能力→价值"对照(现成的售前弹药)

| 层 | 模块 | 主要职责 | 客户价值 |
|---|---|---|---|
| 采集端 | 边缘网关(protocol-router / telegraf / ai-identifier) | 串口/TCP/CAN/Modbus 接入、协议初判、解析、转发、落库 | 统一现场设备接入,降低多协议复杂度 |
| 总线 | EMQX MQTT | 承载数据/状态/告警/错误/识别消息 | 实时性 + 多模块订阅同一数据源 |
| 服务端 | yogee-iot-golang 业务中台 | 台账、端口绑定、监控、告警、健康评分、动态识别、AI 诊断 | 为前端/第三方提供稳定统一数据服务 |
| 存储 | PostgreSQL / TimescaleDB / Redis | 台账/采集/告警/评分/配置持久化;动态映射与运行期缓存 | 历史查询、趋势分析、问题追溯、状态恢复 |
| 运维 | Loki + Grafana | 日志聚合、查询、可视化 | 故障定位与客户验收 |

> 原文第 4 / 6.2 节的模块表本身就带"客户价值/对客户作用"列,是**已经翻译好的"能力→价值"**,可直接喂给 🛠️ Sales Engineer。

## 四、真正"不会变的底座"= 三套契约

"后续拓展底层不变"落到工程上,就是下面三张表必须当成**冻结的接口契约**:拓展永远是"往契约里加条目",而不是"改契约本身"。

### ① MQTT Topic 命名空间(采集链路契约)

| Topic | 语义 |
|---|---|
| `/data/device` | 标准设备数据(主数据流) |
| `/iot/error`(或 `/internal/error`) | 采集 / 解析异常 |
| `/iot/status` | 设备在线状态 |
| `/iot/router/bind` | 动态端口绑定通知 |
| `/iot/identify/request` `…/pending` `…/progress` `…/confirmed` | 未知设备识别四段式流程 |

### ② 核心数据表(业务模型契约)

`devices`(主台账) · `device_alarms`(告警+确认) · `device_registry`(动态注册+字段元信息+端口映射) · `telegraf_device_record`(原始报文+结构化留痕) · `pending_identifications`(待确认) · `device_health_score_rules` / `_history`(评分规则与历史) · `port_cards` / `_template`(端口卡片与模板) · `ship_location_categories` / `ship_system_categories`(船舶位置/系统分类)。

### ③ 对外接口形态(集成契约)

REST(增删改查+落库)· WebSocket(识别事件/评分实时推送)· SSE(AI 诊断逐段流式)· Swagger(联调验收)。

> **边界精确定义**:新增一个设备 = 加一份 Telegraf 解析器 + 一条端口绑定 + 命中一个模板;新增一类健康判断 = 加一条 `device_health_score_rules`;新增一个前端 = 加一个 REST/WS 订阅方。**这些动作全部发生在契约内部——五层管道、那组 Topic、那组表、那四种接口形态本身不动。这才是"底座不变"的范围。**

## 五、两条业务流:差异化卖点与诚实留白

**已知设备(自动接入)**:报文特征命中模板 → 路由到固定 Telegraf 端口 → 预置解析器结构化。**全程不依赖 AI,自动绑定。** 性能与可靠性的基本盘。

**未知设备(辅助接入)**:采样原始报文 → 发 `identify/request` → 本地 Ollama 辅助判断协议/字段 → 生成候选 Telegraf 配置 → 服务端 `pending` 待确认 → **人工确认** → `confirmed` 回传 → 热重载绑定 → 进入标准链路、建立动态台账。

这条 **"AI 辅助识别 + 人工确认 + 热重载即插即用"** 是卖点。但原文有一句必须照搬、不能拔高的诚实留白:

> **"不建议将其定义为完全替代人工协议分析的能力。"**

正确话术是"**降低新增设备接入成本、缩短接入初期分析时间**",human-in-the-loop 是特性不是缺陷。把它说成"AI 全自动接入任何设备"就是给交付埋雷。

**另一条硬留白**:Modbus 复杂字段"需结合厂家寄存器表";第 9.2 节进一步要求客户提供"**完整寄存器表、量纲、倍率、报警位定义和通讯参数**"。→ 这是一条**必须在方案里前置写明的客户输入依赖项**,否则现场实施会卡死在这里且责任边界不清。

## 六、值得注意的技术抉择(解读)

- **用 Telegraf 当协议解析器**:把"写解析代码"降级成"配解析插件",这是"新增设备低成本"承诺的工程支撑。代价是受 Telegraf 解析模型边界约束——强状态/复杂二进制 Modbus 换算场景,AI 生成的配置仍需人工校核。
- **两套 AI,职责不同**:采集侧是**本地 Ollama**(识别未知设备);服务侧是**外部 AI 工作流**(MindDuck/客户指定,做诊断问答,SSE 流式)。售前别混为一谈。
- **Timescale + Redis 分工**:时序落库走 TimescaleDB(趋势/历史/追溯);动态端口映射与识别档案走 Redis(运行期+状态恢复)。健康评分"写历史表 → 可转实时告警/趋势",把评分做成了**预测性维护的数据地基**。

## 七、待补强 / 投标必答的几处(风险提示)

提案时要么主动补强成差异化、要么明确划入边界:

1. **安全是"建议"不是"已交付"**(第 10.2 节全是"应/应该":强口令、TLS、网络隔离、限制 Docker Socket)。→ 可反向操作:把这些做成"开箱即交付的安全基线",立刻变差异化。
2. **可靠性止步于"容器重启恢复"**,未提 HA / 集群 / 消息持久化 QoS / 背压。对**船岸卫星链路间歇中断**——断链期间数据是丢还是 store-and-forward?原文没答,运营域客户必问,需补明确说法。
3. **AI 诊断价值部分依赖外部工作流**(MindDuck/客户),稳定性与效果有一截不在本栈掌控内,报价与 SLA 要把这段责任切清楚。

## 九、底座速记(新人必知)

- **产品**:Yog IOT,船舶运营域(元启晨星)。服务端代号 `yogee-iot-golang` / `yogee-app`。
- **脊椎**:EMQX MQTT 消息总线;一切解耦的根。
- **采集三件套**:protocol-router(协议初判/路由)+ telegraf-collector(解析)+ device-ai-identifier(本地 Ollama 识别)。
- **协议覆盖**:NMEA 0183(AIS/测深/倾角/雾号/自动舵/气象/北斗/罗经/火警/总报警等)、CAN(毫米波雷达 RADAR1-4)、Modbus TCP/RTU(主推进/侧推配电/直流母线/电池箱/航行灯/姿态/能见度/液位等)。
- **部署**:Docker Compose;船端独立 / 岸端集中 / 船岸协同三种形态。
- **不变的底座 = 三套契约**:MQTT Topic 命名空间 + 核心数据表 + REST/WS/SSE/Swagger 接口形态。
- **两条流**:已知设备自动绑定(无 AI);未知设备 AI 辅助 + 人工确认 + 热重载。
- **必须前置的客户输入**:Modbus 厂家寄存器表(量纲/倍率/报警位/通讯参数)。
