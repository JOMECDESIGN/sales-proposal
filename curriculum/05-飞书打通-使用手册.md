# 05 · 飞书打通使用手册(新人视角)

> 前置:本仓库已与飞书三层打通(MCP / 文档同步 / 自动化),配置与凭证见 [`feishu/README.md`](../feishu/README.md)。
> 本篇只讲**新人怎么用**:哪个 Agent 能调飞书、嵌进五段流程哪一步、四个即用动作。

---

## 一、Agent → 飞书能力对照

7 个 Agent 通过 ① 层 MCP 在对话中直接读写飞书。常用对应:

| 链路环节 | 主导 Agent | 在飞书里做什么 | 走哪层 |
|---|---|---|---|
| ① 挖痛点 / 干系人 | 🛠️ Sales Engineer · 🗺️ Account Strategist | 把访谈纪要建成云文档、按干系人维护多维表格 | ① MCP |
| ② 定位 / 赢主题 | 🏹 Proposal Strategist | 执行摘要 / 赢主题落成云文档,放进知识库 | ① MCP + ② 同步 |
| ③ 技术主体 / 架构 | 🛠️ SE · 📐 UX Architect · 🧭 PM | 架构图说明、对比矩阵同步到飞书表 | ① MCP + ② 同步 |
| ④ 计划 / 风险 | 🐑 Project Shepherd | 关键路径、风险登记进多维表格看板 | ③ 自动化 |
| ⑤ 拿单 / 答辩 / 复盘 | 🏋️ Sales Coach | 方案状态卡片播报、复盘归档 | ③ 自动化 |

> 心法不变:**你要练的是"什么时候叫哪个 Agent、怎么喂料、怎么验收"**;飞书只是让产出"长在团队协作里"。

## 二、嵌入五段流程(对照 [04 流程篇](04-流程篇-从挖需求到拿单.md))

- **第①段 挖需求**:访谈后让 SE「把纪要建成飞书云文档」;Account Strategist 把干系人写进多维表格。
- **第②段 定位**:Strategist 产出执行摘要 → 同步进知识库 `售前方案` 空间(② 层)。
- **第③段 技术主体**:方案正文走 Markdown,定期 `sync.sh push` 同步;评审意见在飞书文档里批注、再回流。
- **第④段 计划**:Project Shepherd 把交付物/验收/风险写进销售管道表(`pipeline_base.py`)。
- **第⑤段 拿单**:每次状态变更用 `notify.py` 发卡片到 `#售前播报` 群;复盘文档归档进知识库。

## 三、四个即用动作(复制即用)

**A. 让 Agent 把当前方案建成飞书文档**
> "用 lark MCP 把《XX 项目方案》执行摘要建成一篇飞书云文档,放进知识库 `售前方案` 空间。"

**B. 批量同步教材 / 方案到飞书**
```bash
set -a && source feishu/.env && set +a
feishu/02-doc-sync/sync.sh push curriculum     # 只同步教材分组
```

**C. 播报一次方案状态到群**
```bash
python feishu/03-automation/notify.py \
  --title "XX项目方案" --status 答辩中 --owner 张三 --note "周五现场演示"
```

**D. 把一条机会写进销售管道看板**
```bash
python feishu/03-automation/pipeline_base.py add \
  --name "XX项目" --customer "XX公司" --stage 撰写中 --owner 张三 --amount 120 --close-date 2026-07-31
```

## 四、常见坑(踩过就记住)

- **机器人没进群** → 发卡片必失败。先把应用机器人拉进群,再 `feishu-cli chat list` 取 `chat_id`。
- **改了权限没发版本** → scope 不生效。开放平台改完权限**必须发布新版本**。
- **CI 没动静** → `sync-map` 里没填 `doc_id` 的条目在 update-only 下会被跳过;首次入库需人工一次(见 [`feishu/02-doc-sync/README.md`](../feishu/02-doc-sync/README.md))。
- **凭证别入库** → 只进 `feishu/.env` / GitHub Secret,已 gitignore。
